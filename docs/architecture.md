# Схема архитектуры

![Схема архитектуры](architecture.svg)

## Компоненты
- `Docker Compose` оркестрирует локальный и продакшен-стек приложения
- `api` - контейнер с Django API и DRF
- `db` - контейнер PostgreSQL, который хранит основное состояние постов
- `redis` - контейнер Redis, который хранит кеш сериализованных постов
- `Nginx` - опциональный внешний reverse proxy для production-сценария, сейчас
  в проекте представлен шаблоном `deploy/nginx.template.conf`, а не отдельным
  сервисом в compose-файлах

## Поток запроса `GET /api/v1/posts/{id}/`
1. Клиент отправляет запрос напрямую в API или через внешний `Nginx`.
2. Django сначала проверяет наличие ключа `post:{id}` в Redis.
3. Если ключ найден, ответ возвращается из кеша.
4. Если ключ не найден, Django читает пост из PostgreSQL.
5. Данные сериализуются и сохраняются в Redis.
6. Ответ возвращается клиенту.

## Поток запроса `PATCH/PUT/DELETE /api/v1/posts/{id}/`
1. Django изменяет или удаляет запись в PostgreSQL.
2. Ключ поста в Redis удаляется.
3. Следующий `GET` снова прогревает кеш из базы.

## Почему выбран такой подход
- PostgreSQL остаётся источником истины
- Redis используется только как быстрый слой чтения
- реализация простая и прозрачная для отладки
- не нужны фоновые задачи и дополнительная координационная логика
- легко точечно инвалидировать кеш при изменении записи
